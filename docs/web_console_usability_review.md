# 本地工作台可用性诊断报告（面向非技术用户）

> 目标读者：完全不懂编程的个人用户。
> 诊断方式：用 Playwright（Chromium）在隔离临时数据库上启动现有 FastAPI 工作台
> （`python -m wechat_article_scheduler.cli serve`），在三种视口宽度截图并抓取 DOM。
> 复跑命令：`.venv/bin/python tools/browser_automation/ui_review.py --seed 5`
> 截图与结构快照：`docs/reports/ui_review/`（含 `dom_snapshot.md`）。
> 本报告只做诊断与规划，不实现大功能。对应的执行计划见 `docs/rounds.md` Phase 6 / Phase 7 / Phase 8（Round 19–38）。

## 0. 桌面优先结论

这个工具不应默认做成手机看板。它的主要使用场景是个人在电脑浏览器里管理本地文章、排期、任务队列和运行日志，核心价值来自信息密度、批量扫读、表格比较、状态定位和安全确认。手机/平板只需要兼容：页面不能横向溢出，关键按钮要能点，文字要能读，但窄屏不是默认布局目标。

后续路线图按 **Desktop-first local workbench** 调整：优先设计桌面首屏信息架构、左侧导航/顶部状态、主内容区、任务队列表格、文章列表、事件日志和高级调试折叠区。**Responsive compatibility** 作为每轮的兼容验收边界，不再作为牺牲桌面端信息密度的主目标。

## 0.1 普通用户模式结论

当前页面最大的体验问题不是“功能少”，而是默认把工程师和 Agent 才需要的信息放到了普通用户面前。后续应把工作台分成三层：

| 层级 | 目的 | 默认可见 | 示例 |
|---|---|---|---|
| 普通视图 | 让非技术用户知道是否安全、该点哪一步、结果是什么 | 是 | “当前是演练，不会真的发到公众号”“已收录 5 篇新文章” |
| 详情视图 | 追踪某篇文章或某个发布任务 | 用户点击后 | 文章标题、计划时间、失败原因、可做操作 |
| 高级/调试视图 | 给 Agent/开发者排错 | 否，需开关 | 数据库路径、原始 JSON、内部 id、payload |

普通视图只回答四个问题：

1. 现在安全吗：会不会真的发布到公众号。
2. 我现在该做什么：按 1、2、3 步完成找文章、排时间、执行到点文章。
3. 刚才做成了吗：用人话反馈本次操作结果。
4. 出错了怎么办：给下一步修复建议，而不是展示内部字段。

`processed`、`skipped_future`、`publish_jobs`、`payload_json`、数据库路径、原始 JSON、英文状态枚举等内容不应出现在普通视图。它们应进入“高级信息”开关或 `/debug` 页面。

DOM 抓取结论（见 `dom_snapshot.md`）：

|---|---|---|---|---|
| desktop_1280 | 否 | 1280 / 1280 | 否 | 1280 / 1280 |
| tablet_768 | **是** | 809 / 768 | 否 | 768 / 768 |
| mobile_375 | **是** | 645 / 375 | 否 | 375 / 375 |

> 修复前平板与手机均出现横向滚动条 —— 这是真实布局缺陷，由两个原因叠加：
> （1）“安全状态”卡片直接展示数据库**绝对路径**这种超长不可断行字符串，把卡片撑破；
> （2）“发布队列”“事件日志”两张表格在窄屏下内容宽度超过容器，把整页顶宽。
> 本轮已做最小修复（见第 7 节）。窄屏表格更友好的卡片化重排留到 Round 26。

### 1.2 首屏区块清单（自上而下）

1. **页头**：标题“微信公众号文章调度器基础工作台” + 英文徽章 `mode: mock`。
2. **顶部提示条**：固定显示“页面已加载：当前为本地轻量工作台”。
3. **安全状态**：6 张卡片 —— 运行模式 `mock`、`DRY_RUN` `off`、真实发布开关 `publish disabled`、安全提示 `mock`、最近操作 `未执行`、SQLite 数据库（**完整绝对路径**）。
4. **操作区**：4 个按钮 —— 扫描 inbox、生成计划、执行到期任务、刷新状态。
5. **结果区**：直接罗列接口返回的原始字段（`wechat_mode`、`dry_run`、`wechat_enable_publish`、`database`），下方“下一步建议”，再加一个“查看原始 JSON”折叠块。
6. **状态卡片**：文章 `imported`/`published`、任务 `pending`/`done`/`failed`/`running`（标签全是英文状态词）。
7. **发布队列 / 最近任务**：表格列 ID / 标题 / 计划时间 / 状态 / 重试。
8. **事件日志**：表格列 时间 / 类型（`plan_created`、`scan_imported`）/ 对象（`article #5`）/ 摘要（直接是 `chapter_006_cn.md`、`payload_json`）。
9. **文档入口**：README / 开发路线图 / Web 控制台设计。

## 5. 术语对照表（后续 UI 文案标准）

> 后续所有面向用户的文案，统一以“普通说法”为准；英文/内部字段名只允许出现在“高级 / 调试”区。

| 内部名 / 现状 | 普通用户说法 | 备注 |
|---|---|---|
| inbox | 收件箱（待处理文章） | 放文章的地方 |
| scan | 扫描收件箱 | 把新文章读进来 |
| plan | 安排发布时间 | 给文章排队 |
| run-once / 执行到期任务 | 发布到点的文章 | 把到时间的文章发出去（演练或真实） |
| publish_jobs | 发布任务 / 排期表 | 已排队等待发布的文章 |
| imported | 已收录 | 文章已读入系统 |
| published | 已发布 | 文章已发出 |
| pending | 待发布（排队中） | 还没轮到 |
| running | 发布中 | 正在发 |
| done | 已完成 | 发布成功 |
| failed | 失败 | 发布没成功 |
| skipped_future | 未到时间 | 已排期，时间还没到 |
| skipped_max_retries | 重试用尽 | 多次失败后不再尝试 |
| dry_run | 演练 / 空跑 | 假装发布，不真的发出去 |
| mock | 演练模式 | 本地模拟，不联网、不会真发 |
| real | 真实模式 | 会真的连接微信公众号 |
| WECHAT_ENABLE_PUBLISH | 是否允许真的发布 | 安全开关 |
| adapter_mode | 运行通道 | 演练 / 真实 |
| retry_count | 已重试次数 | 失败后又试了几次 |
| media_id | 微信素材编号 | 草稿在微信侧的内部编号 |
| digest / summary | 文章摘要 | 简介 |
| event_type | 操作类型 | 发生了什么 |
| entity_type / entity_id | 关联对象 | 这条记录对应哪篇文章/任务 |
| payload_json | 详细信息（原始数据） | 调试用 |
| SQLite database | 本地数据文件 | 一般用户无需关心路径 |

## 6. 信息架构建议

目标：**首屏只放高频内容，技术细节折叠或下沉**。

- **首屏（人人都看）**：
  1. 一句话状态（安全/是否会真发/待办数量）；
  2. 三步操作引导（找文章 → 排时间 → 发布/演练）；
  3. 最近一次操作的人话结果；
  4. “待发布文章”简明列表（桌面优先表格，窄屏顺序堆叠或容器内滚动）。
- **次屏 / 折叠（想了解的人看）**：
  - 发布队列详情、事件日志（人话摘要）。
- **高级 / 调试区或单独 `/debug` 页（开发者看）**：
  - 数据库路径、原始 JSON、内部统计字段、事件 `payload_json` 原文、英文状态枚举。

落地编排：普通用户工作台原则与 Playwright 视觉基线（R19）→ 术语人话化（R20）→ 普通视图信息减法（R21）→ 三步操作主流程（R22）→ 人话反馈系统（R23）→ 空状态与首次使用（R24）→ 安全发布护栏（R25）→ 桌面主布局（R26）→ 文章列表普通化（R27）→ 发布队列普通化（R28）→ 事件日志时间线（R29）→ 高级信息开关（R30）→ 帮助与解释系统（R31）→ 错误与恢复指引（R32）→ 桌面效率优化（R33）→ 窄屏兼容验收（R34）→ Playwright E2E 可用性基线（R35）→ 非技术用户走查报告（R36）→ Web 控制台 MVP 收口（R37）→ 后续功能接入规范（R38）。

## 7. 本轮已做的最小安全修复

仅做低风险、不改业务语义的 CSS 修复（见 `src/wechat_article_scheduler/web/app.py` 样式块）：

1. 为状态卡片数值加 `overflow-wrap: anywhere; word-break: break-word;`，让数据库长路径在卡片内换行，不再撑破卡片（手机 scrollWidth 645 → 437）。
2. 为 `#jobsArea`、`#eventsArea` 两个表格容器加 `overflow-x: auto`，让宽表在容器内横向滚动而不顶宽整页（手机/平板 scrollWidth 最终回落到等于视口宽度，三视口横向溢出全部消除）。

验证：`.venv/bin/python tools/browser_automation/ui_review.py --seed 5` 三视口均 `overflow=False`。
更友好的普通视图、桌面队列表格、数据库路径下沉、术语人话化、高级信息开关与窄屏兼容等较大改动均按 Round 19–38 推进，不在本轮提前做完。
