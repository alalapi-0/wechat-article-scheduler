# Web 控制台设计（轻量）

## 目标

在不引入 React/Vue 的前提下，提供本地可用、可审计的操作面板。

## 当前现状

- 已实现：`FastAPI + 内嵌 HTML`，支持状态查看与 scan/plan/run-once 触发
- 部分实现：文章、任务、事件基础查询
- 未实现：鉴权、多用户、复杂筛选、审批工作流

## 设计原则

- 继续使用服务端模板或极简原生 JS
- Web 仅作为 CLI 的可视化壳，业务逻辑仍在 Python 服务层
- 默认只允许本机访问（`127.0.0.1`）

## 桌面优先产品原则

本控制台默认形态是 **Desktop-first local workbench**：主要服务电脑浏览器中的本地发布工作台，而不是手机看板。后续 Web 轮次应优先保证桌面端的信息密度、批量处理效率和可定位性。

- 桌面推荐布局：左侧导航或清晰分区导航 + 顶部安全/运行状态 + 主内容区（概览、三步操作、文章列表、发布队列）+ 次级内容区（事件日志、帮助）+ 默认折叠的高级/调试信息。
- 桌面优先内容：任务队列表格、文章列表、事件日志应适合 1280px 左右的电脑浏览器快速扫读，保留标题、计划时间、状态、重试、最近结果等高频信息。
- 窄屏边界：手机/平板只做 **Responsive compatibility**，验收目标是不横向溢出、关键按钮可点、页面可读；允许顺序堆叠或容器内滚动，但不能为了手机卡片化牺牲桌面端信息密度。
- 用户友好仍然保留：术语用人话，内部字段、绝对路径、原始 JSON 默认下沉到高级/调试区。

## 普通用户视图原则

默认页面首先服务“完全不懂编程的人”。普通视图只回答四个问题：

1. 现在安全吗：当前是否只是演练，是否会真的发布到公众号。
2. 我现在该做什么：按“找文章 → 安排时间 → 执行到点文章”三步走。
3. 刚才做成了吗：用一句话说明扫描、排期、执行的结果。
4. 出错了怎么办：说明原因、影响和下一步，而不是显示内部字段。

普通视图不展示：数据库绝对路径、原始 JSON、`payload_json`、`processed`、`skipped_future`、`retry_count`、内部 id、hash、文件绝对路径、英文状态枚举、`.env` 或任何密钥相关内容。

## 信息分层

| 层级 | 面向对象 | 默认展示 | 可出现内容 | 不应出现内容 |
|---|---|---|---|---|
| 普通视图 | 非技术用户 | 是 | 安全状态、三步操作、待办数量、最近结果、人话错误 | 原始 JSON、数据库路径、内部字段、英文枚举 |
| 详情视图 | 想追踪具体文章/任务的用户 | 用户点击后 | 文章标题、合集、状态、发布时间、失败原因、可做操作 | token、secret、cookie、未解释的 payload |
| 高级/调试视图 | Agent、开发者、排错者 | 否，需开关或单独入口 | 数据库路径、内部 id、原始响应、事件 payload、内部统计 | 任何密钥、access_token、AppSecret、cookie |

后续新增任何 Web 功能时，都必须先定义：普通视图显示什么、详情视图显示什么、高级信息归到哪里、错误如何人话化。

## 推荐桌面首屏

1. 顶部安全条：一句话说明“当前是演练，不会真的发到公众号”或真实模式风险。
2. 三步主流程：找文章、安排时间、执行到点文章；刷新状态放在次要位置。
3. 最近结果：显示最近一次操作做成了什么，以及下一步建议。
4. 待办摘要：今天/未来待发布文章数量、失败数量、需要处理的问题。
5. 详情入口：文章列表、发布队列、事件日志、高级信息。

## 计划接口

- 只读：状态、文章、任务、事件
- 受控写：scan、plan、run-once（后续增加审批闸门）
- 明确禁止：直接触发无审计的真实发布批处理
