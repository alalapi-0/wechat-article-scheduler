# 微信公众号草稿分批定时发布浏览器测试

Status: Draft for screenshot-guided validation

本文件用于测试 Cursor Agent 能否通过 `wechat-chrome-session`，在用户已登录的微信公众号后台中，为指定草稿设置固定字段，并把后台定时流程推进到微信要求的人工安全验证节点。

这不是官方发布 API 测试。当前账号没有 `freepublish/submit` 权限；这里测试的是微信公众号后台页面提供的定时能力。

2026-06-07 实机验证结论：页面接管可用，但发布/定时操作可能触发管理员扫码或手机确认。该验证属于微信平台的风险操作保护，不能由 Agent 代替、读取二维码或绕过，因此本流程不是无人值守发布。

## 一、Cursor 运行要求

- 使用普通前台 Agent，禁止 Multitask 和子 Agent。
- 只使用 `wechat-chrome-session`。
- 禁止 `browser_tabs`、普通 `chrome-devtools`、`playwright --isolated`。
- 禁止 `new_page` 和 `navigate_page`；接管现有 `mp.weixin.qq.com` 标签页。
- 不读取或输出 cookie、token、URL query、Authorization、二维码内容。
- 出现二维码、管理员确认、手机确认或安全验证时立即停止，由用户本人完成。
- 每次页面状态变化后重新 `take_snapshot`，根据当前 DOM 定位控件；截图只用于解释页面，不得盲按固定坐标。

## 二、测试参数

执行前由用户填写：

```text
测试草稿标题：
固定合集：
是否开启留言：
是否仅粉丝可留言：
是否允许推荐：
是否通知用户：
是否显示封面：
测试定时时间（Asia/Shanghai）：
```

单篇测试时间必须是未来时间，并满足微信页面显示的最小提前量。若页面提示时间无效、数量受限或需要管理员验证，立即停止。

## 三、截图说明占位

用户将按操作顺序提供截图。Cursor Agent 应把截图与实时 DOM 共同使用：

| 编号 | 页面/动作 | 需要从截图补充的内容 |
|---|---|---|
| S1 | 首页进入草稿箱 | 左侧菜单名称、草稿箱入口 |
| S2 | 草稿列表 | 搜索框、目标草稿卡片、编辑入口 |
| S3 | 草稿编辑页 | 合集、留言、仅粉丝留言、封面设置位置 |
| S4 | 发布设置 | 推荐、通知等开关名称和默认状态 |
| S5 | 发布/定时入口 | 发布按钮、定时选项入口 |
| S6 | 日期时间弹窗 | 日期、小时、分钟、时区和限制提示 |
| S7 | 最终确认弹窗 | 会产生真实定时任务的最终按钮 |
| S8 | 定时成功页/列表 | 成功提示、目标时间、取消或修改入口 |

若截图和当前页面 DOM 不一致，以当前 DOM 为准，并报告 `UI_CHANGED`，不得猜测点击。

## 四、阶段 1：只读定位

依次执行：

1. `list_pages` 找到 `mp.weixin.qq.com`。
2. `select_page` 选择已登录公众号后台。
3. `take_snapshot` 和 `take_screenshot`。
4. 确认账号后台可见，且没有登录、验证码、风控或管理员验证页面。
5. 进入内容管理/草稿箱。
6. 只定位用户指定的测试草稿，不进入其他草稿。
7. 使用标题精确匹配；存在重名时列出脱敏摘要并请用户指定，不自行选择。

阶段 1 不修改任何字段。

## 五、阶段 2：单篇发布前准备并保存草稿

对测试草稿逐项核对：

1. 标题与用户指定标题完全一致。
2. 正文、摘要和封面存在，不修改文章内容。
3. 固定合集设置为用户指定合集。
4. 留言、仅粉丝留言、推荐、通知、封面显示按测试参数设置。
5. 填写目标定时时间；若页面只有进入发布流程后才显示时间控件，可以进入发布设置，但不得最终确认。
6. 读取页面回显，核对日期、时间和时区。
7. 返回可保存状态并点击“保存草稿”。
8. 回到草稿箱，重新打开同一草稿。
9. 核对合集、开关、封面和目标定时时间是否仍然存在。
10. 停在已保存草稿页面，不点击正式发表。

停止后输出：

```text
单篇定时预演报告

- 草稿标题：<标题>
- 固定合集：<合集>
- 留言：on/off
- 仅粉丝留言：on/off
- 推荐：on/off
- 通知：on/off
- 显示封面：on/off
- 目标时间：YYYY-MM-DD HH:mm Asia/Shanghai
- 草稿已保存：yes/no
- 重新打开已核验：yes/no
- 定时时间随草稿保存：yes/no/not_supported
- 当前页面：已保存草稿/发布设置最终确认前
- 已创建真实定时任务：no
- 页面警告：none/<摘要>
- 结果：READY_FOR_CONFIRMATION/BLOCKED
```

同时提供保存后重新打开的截图。若定时时间没有持久化，不得声称草稿已经完成定时，只记录目标时间供用户正式发表前回填。

## 六、阶段 3：用户最终发表与安全验证

只有用户发送如下明确批准后才可继续：

```text
批准测试定时：
草稿标题=<精确标题>
目标时间=<YYYY-MM-DD HH:mm Asia/Shanghai>
允许点击一次最终定时确认。
```

Agent 的准备工作在阶段 2 已结束。用户本人负责：

1. 打开 Agent 已准备好的草稿。
2. 核对标题和目标时间；若时间未随草稿保存，按 manifest 回填。
3. 点击正式发表/定时确认。
4. 完成二维码、手机确认或管理员验证。

用户完成后，可让 Agent 只读核验待发布/定时列表中的标题与目标时间。

若按钮点击后无明确成功反馈，不得重复点击。报告 `RESULT_UNCERTAIN` 并交给用户检查。

成功报告：

```text
真实定时测试报告

- 草稿标题：<标题>
- 目标时间：<时间>
- 后台状态：<页面显示状态>
- 成功提示：<脱敏摘要>
- 定时列表可见：yes/no
- 重复点击：0
- 人工安全验证：required/completed/not_shown
- 截图证明：<工具返回标识或安全路径>
- 结果：PASS/RESULT_UNCERTAIN/BLOCKED
```

## 七、阶段 4：生成批量排期清单

单篇真实测试 PASS 后，先生成 manifest，不立即批量点击。

默认建议方案为每周 35 篇、每天 5 篇：

```text
时区：Asia/Shanghai
每日时段：09:00、12:00、15:00、18:00、21:00
每周天数：7
每周最多：35
排序：按用户确认的草稿顺序
下周续排：从上周最后一篇之后继续，不重复
```

manifest 每行必须包含：

```text
序号 | 精确标题 | 目标时间 | 固定合集 | 留言 | 仅粉丝留言 | 推荐 | 通知 | 显示封面
```

生成时检查：

- 标题不重复。
- 时间严格递增。
- 不与后台已有定时任务冲突。
- 不超过页面显示的每日数量或时间限制。
- 所有目标时间均在未来。
- 本周已处理或已定时的草稿不得再次加入。

## 八、阶段 5：批准后的批量设置

只有用户明确回复下面格式，才允许按 manifest 批量设置：

```text
批准批量定时 manifest：
总数=<N>
首篇=<标题>
首个时间=<时间>
末篇=<标题>
最后时间=<时间>
允许按已展示 manifest 逐篇点击最终定时确认。
```

批量执行规则：

- 仅操作已批准 manifest 中的草稿。
- 每篇最终确认前再次核对标题和时间。
- 每篇成功后记录 proof，再进入下一篇。
- 不确定是否成功时停止整个批次，不重复点击。
- 页面结构变化、验证码、管理员确认、平台限额、账号警告、重名或时间冲突时立即停止。
- 支持从最后一篇明确成功的记录继续，禁止从头重跑。
- 每次出现微信扫码/手机确认都必须由用户本人处理。若每篇均触发验证，则该批次只能采用“Agent 填写一篇 → 用户验证 → Agent 核验 → 下一篇”的半自动模式。
- 不得通过关闭浏览器安全提示、隐藏自动化标记、复用二维码、提取确认链接或修改请求来规避验证。

## 九、无人值守能力结论

Agent 可以无人值守完成已批准 manifest 中的“发布前准备并保存草稿”，前提是保存草稿本身不触发验证。

正式发表仍由用户执行：

- 若定时时间可随草稿保存，用户主要负责最后发表和扫码。
- 若定时时间不能随草稿保存，用户发表前还需按 manifest 回填时间。
- Agent 不代替二维码、手机确认或管理员验证。
